# Fixture Logging Reference

This document describes:

- runtime `PasteAsMd` debug logs
- how to turn those logs into fixture inputs
- fixture metadata scenarios
- harness-generated outputs and cleanup

## 1) Runtime debug logs (`PasteAsMd.ahk`)

When `PasteMd.DEBUG_PASTE_MD := true`, `PasteAsMd.ahk` writes UTF-8 debug logs
to `PasteMd.DEBUG_PASTE_MD_LOG` (default: `A_ScriptDir\PasteAsMd_debug.log`).

The logger uses `PasteMd._DbgSection(...)` and writes pipeline stages such as:

- `1. plain (A_Clipboard minus CR)`
- `2. cfHtml (raw full payload)`
- `3. htmlFrag (CF_HTML fragment)`
- `2b. cfHtml offsets`
- `3. htmlPrep (after _PreprocessHtml)` (HTML path)
- `3b. md (no HTML tags → plain text path)` (when applicable)
- `4. mdRaw (pandoc output)` (when applicable)
- `5. md (after CleanMarkdown)` (when applicable)
- `5c. expected list start (ordered-list fix)`
- `5d. md (after RestoreOrderedListStart)`
- `5e. md (after SHOW_POSTER replacement)` (quoted mode)
- `6. FINAL md (pasted)`

## 2) Converting a runtime log into a fixture

1. Capture a real run with debug enabled.
2. Copy/rename the log into `ahk-test/` as `PasteAsMd_<NAME>.log`.
3. Keep at minimum sections:
   - `1. plain (A_Clipboard minus CR)`
   - `2. cfHtml (raw full payload)`
4. Add expected output file(s):
   - default: `PasteAsMd_<NAME>.expected.md`
   - metadata scenarios: `PasteAsMd_<NAME>.<CASE>.expected.md`

The fixture harness decodes those sections and calls
`PasteMd._ConvertFromCaptured(...)` directly.

## 3) Metadata scenarios (in fixture `.log` header)

Metadata is read from non-blank header lines after line 1 and before the first
`===` section marker.

Each metadata line is one scenario with comma-separated `key:value` pairs.

Supported keys:

- `case` (required on each metadata line; for the default unsuffixed scenario, use no metadata lines)
- `prompt` (optional): `CANCEL` or integer `> = 1`
- `expectAbort` (optional): `0` or `1` (default `0`)

Unknown keys fail parsing.

Examples:

```text
case:renumber3,prompt:3
case:cancel,prompt:CANCEL,expectAbort:1
```

If no metadata lines exist, the harness runs one default scenario.

## 4) Harness CLI controls

`test-paste-md-fixtures.ahk` supports:

- `/ls`
- `/fixture:<n>`
- `/fixtureOutputLogs:0|1` (default `0`)

`/fixtureOutputLogs` is CLI-only (not metadata-driven).
`/fixtureOutputLogs:1` by itself runs all fixtures and writes per-scenario
`.fixture.log` outputs.

## 5) Generated artifacts and cleanup

Generated by fixture harness:

- `test-paste-md-fixtures.log`
- `PasteAsMd_*.actual.md`
- `PasteAsMd_*.fixture.log` (only with `/fixtureOutputLogs:1`)

Cleanup script: `test-clean.ahk`

`test-clean.ahk` deletes generated test logs:

- `test-*.log`
- `PasteAsMd_*.actual.md`
- `PasteAsMd_*.fixture.log`

`test-clean.ahk` does not delete:

- source fixtures: `PasteAsMd_*.log`
- expected outputs: `*.expected.md`

## 6) Per-scenario fixture output logs

When `/fixtureOutputLogs:1` is set, the harness writes replay-style logs
mirroring `PasteAsMd` stage labels for each scenario:

- default scenario: `PasteAsMd_<NAME>.fixture.log`
- case scenario: `PasteAsMd_<NAME>.<CASE>.fixture.log`

These logs are diagnostic outputs only; fixture inputs remain `PasteAsMd_*.log`.
